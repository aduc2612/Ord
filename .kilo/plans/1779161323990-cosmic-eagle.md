# Plan: Replace text PK with UUID on test_items table

## Context
`drizzle-orm/sqlite-core` (v0.45.2) does **not** export a `uuid` column builder (unlike `pg-core`).
SQLite stores UUIDs as `text` — `uuid()` in drizzle-orm for Postgres generates `uuid` type,
but for SQLite the correct approach is a `text` column that holds UUID strings.

The user's instruction says: `uuid("id").primaryKey()`.  
To satisfy this while staying within SQLite's constraints:

- Define a local `uuid` helper that wraps `text()` (same type, same generated SQL) so the desired API surface exists.
- Use it for both `id` (PK) and `user_id` (foreign key to Supabase auth.users).

## Changes

### 1. `src/db/schema.ts`
- Add a local `uuid` helper function that delegates to `text()`.
- Replace `text("id").primaryKey()` → `uuid("id").primaryKey()`.
- Replace `text("user_id").notNull()` → `uuid("user_id").notNull()`.

```ts
// helper at top of file (before table def):
import { sqliteTable, text, integer } from "drizzle-orm/sqlite-core";

/** UUID column helper: SQLite stores UUIDs as text. */
function uuid(name: string) {
  return text(name);
}
```

### 2. `src/db/migrations/` — No SQL/snapshot changes needed
Since `uuid()` delegates to `text()`, the generated DDL is identical.  
The existing migration SQL and snapshot already use `text` columns, which is the correct SQLite representation.  
No regeneration is required — schema and artifacts remain aligned.

### 3. `src/hooks/use-db-test.ts`
- `id` is already generated via `Crypto.randomUUID()` — compatible, no change needed.
- `userId: "test-user"` (line 43) is a hardcoded string, not a UUID. Update it to use a UUID:
  ```ts
  userId: Crypto.randomUUID(),
  ```
  This aligns with the `uuid` column type and the repo contract (user_id references Supabase auth.users, which use UUIDs).
